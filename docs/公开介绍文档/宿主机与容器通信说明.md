# 宿主机与容器通信说明

## 1. 运行分工

当前标准部署方式中：

1. 容器内运行上层客户端。
2. 宿主机运行 `nuwax_robot_bridge`。

两者职责固定如下。

### 容器内负责

1. 上层会话和界面。
2. 工具调用与结果消费。
3. 用户交互与编排逻辑。

### 宿主机负责

1. 连接机器人 SDK、DDS 和传感器。
2. 执行运动、导航、感知、记忆等能力。
3. 对外暴露 MCP、HTTP、事件流和制品通道。

## 2. 通信方向

当前标准方向只有一条：

> 容器主动访问宿主机。

不是宿主机反向连接容器。

推荐访问地址如下：

```text
http://host.docker.internal:8766
```

## 3. 标准通信接口

容器访问宿主机时主要使用下面四类接口：

1. `POST /mcp`
   工具调用主入口。
2. `GET /api/*`
   调试和补充接口。
3. `GET /events/*`
   任务进度、状态变化和系统事件订阅。
4. `GET /artifacts/*`
   图片、快照和其他制品下载。

## 4. 鉴权方式

容器访问宿主机网关时，需要带上 Bearer Token。

例如：

```text
Authorization: Bearer agent-robot-bridge-token
```

宿主机同时还会校验来源网段，因此除了 token 之外，还需要保证容器网段已加入宿主机允许列表。

## 5. 容器访问宿主机的关键前提

如果容器使用 Docker 运行，必须加上：

```text
--add-host=host.docker.internal:host-gateway
```

否则容器通常无法解析 `host.docker.internal`。

## 6. 外部 MCP 接入方式

如果容器镜像没有稳定的 MCP 配置页面，当前推荐使用仓库内的外部 MCP 注入脚本。

这套方案的特点是：

1. 不依赖手改数据库。
2. 配置文件可持久化。
3. 容器重启后仍可保留外部 MCP 配置。

默认配置文件路径为：

```text
/var/lib/docker/volumes/nuwax_home/_data/.nuwaxbot/mcp/external_servers.json
```

典型内容如下：

```json
{
  "mcpServers": {
    "nuwax_robot_bridge": {
      "url": "http://host.docker.internal:8766/mcp",
      "headers": {
        "Authorization": "Bearer agent-robot-bridge-token"
      }
    }
  }
}
```

## 7. 推荐理解方式

可以把这套通信关系理解为：

1. 容器是调用方。
2. 宿主机是执行方。
3. 机器人能力只在宿主机生效。

只要这个边界不被打破，后续无论增加机器人、增加相机、增加地图系统，还是升级视觉与记忆链路，整体通信方式都不需要重做。
