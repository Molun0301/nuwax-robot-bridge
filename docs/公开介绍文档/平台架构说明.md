# 平台架构说明

## 1. 项目定位

`nuwax_robot_bridge` 是部署在机器人宿主机上的具身能力运行时。

它不负责上层对话和会话，而是负责：

1. 连接机器人和底层 SDK。
2. 采集状态、图像、地图和位姿。
3. 执行运动、导航、感知、记忆和音频等能力。
4. 通过统一接口把这些能力暴露给上层系统。

## 2. 分层结构

整体结构可以理解为四层：

1. 机器人连接层
   负责机器人 SDK、传感器和硬件能力接入。
2. 平台运行时层
   负责任务、事件、状态、安全和能力编排。
3. 能力服务层
   负责感知、地图、定位、导航、记忆、音频等正式能力。
4. 对外网关层
   负责通过 `MCP + HTTP + 事件流 + 制品下载` 对外暴露能力。

## 3. 典型调用链路

一个典型调用的流向如下：

1. 容器内上层系统发起工具调用。
2. 宿主机网关收到请求。
3. 运行时把请求路由到具体能力服务。
4. 服务通过机器人入口和 provider 执行实际动作或采集数据。
5. 结果通过标准契约返回给网关。
6. 网关再把结构化结果返回给上层系统。

如果结果中包含图片、日志、快照等大对象，则通过制品通道提供下载地址。

## 4. 当前核心能力

当前主线已经具备下面这些能力模块：

1. 机器人状态与安全状态采集。
2. 图像采集与观察记录。
3. 本地视觉检测与云端语义识别。
4. 关键帧驱动的视频感知运行时与到点 burst 复核。
5. 地图、定位、导航与探索。
6. 语义空间记忆闭环：命名地点、对象实例、观察事件与到点复核。
7. 任务管理、事件流和制品管理。

## 5. 视觉与记忆链路

当前视觉链路采用“双识别”结构：

1. 本地识别默认优先使用 `YOLO26n + TensorRT`，负责稳定、实时的目标检测。
2. 云端识别负责开放词表语义、关系描述和细粒度补充。
3. 平台默认不是“每帧全量重检测”，而是“关键帧驱动 + 到点 burst 复核”。
4. 两者合并后写入统一感知结果。
5. 感知结果不会把整张 SLAM 地图写入向量库，而是先投影到 `map frame`，再生成对象实例和观察事件。
6. 记忆库当前按 `place_nodes / object_instances / episodic_observations` 三类集合组织。
7. 导航层消费 inspection pose，而不是直接消费对象中心点。

当前记忆向量链路默认配置如下：

1. 文本向量默认使用 `sentence-transformers:BAAI/bge-m3`。
2. 图像向量默认使用 `transformers-clip:openai/clip-vit-base-patch32`。
3. 如果宿主机缺少相关依赖或首次下载失败，运行时会自动回退到轻量模式，并把原因写入日志。

当前记忆侧的三层对象职责是：

1. `ObservationEvent`
   表示一次带空间锚定的原始观察事实。
2. `SemanticInstance`
   表示跨帧关联后的持续对象实体。
3. `SemanticMemoryEntry`
   表示面向上层消费的摘要条目。

## 6. 扩展原则

平台的扩展方式固定如下：

1. 新机器人通过 `drivers/robots/<robot>` 接入。
2. 新传感器和新地图来源通过 `adapters/` 或 `providers/` 接入。
3. 新视觉模型通过统一后端接口接入。
4. 上层工具接口尽量不因底层模型或硬件变化而改变。

## 7. 对外接口

平台当前对外主要暴露四类接口：

1. `MCP`
   工具调用主入口。
2. `HTTP`
   调试、状态查询和补充调用入口。
3. `SSE/WebSocket`
   任务进度和事件订阅入口。
4. `artifacts`
   图片、快照和其他制品下载入口。
