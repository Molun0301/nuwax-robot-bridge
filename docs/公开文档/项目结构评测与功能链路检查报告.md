# nuwax_robot_bridge 项目结构评测与功能链路深度检查报告

> 评测日期：2026-04-10
> 项目版本：0.9.0（MCP 协议版本 2025-11-25）
> 目标机器人：Unitree Go2（含 G1 骨架支持）

## 一、项目架构总览

### 1.1 分层架构

```
┌─────────────────────────────────────────────────────┐
│                   网关层 (gateways)                   │
│  HTTP API / MCP JSON-RPC / SSE / WebSocket / Relay  │
├─────────────────────────────────────────────────────┤
│                   服务层 (services)                   │
│  Navigation / Mapping / Localization / Perception   │
│  Memory / Audio / Observation / Artifact / RobotState│
├─────────────────────────────────────────────────────┤
│                   核心层 (core)                       │
│  EventBus / SafetyGuard / TaskManager / StateStore  │
│  ResourceLock / CapabilityRegistry                   │
├─────────────────────────────────────────────────────┤
│                   驱动层 (drivers)                    │
│  Go2 Assembly / DataPlane / DDS / Providers         │
│  Navigation Backend / Frontier Exploration           │
├─────────────────────────────────────────────────────┤
│                   契约层 (contracts)                  │
│  Geometry / Maps / Navigation / Memory / Events     │
│  Capabilities / Skills / RuntimeViews / SpatialMemory│
└─────────────────────────────────────────────────────┘
```

### 1.2 模块职责

| 模块 | 职责 | 文件数（约） |
|------|------|-------------|
| `gateways/` | HTTP/MCP/SSE/WS/Relay 网关接入 | 12 |
| `services/` | 导航/建图/定位/感知/记忆/音频等业务服务 | 25+ |
| `core/` | 事件总线/安全守卫/任务管理/状态存储/资源锁 | 7 |
| `drivers/` | Go2/G1 机器人驱动、DDS、数据面 | 30+ |
| `contracts/` | 数据契约与类型定义 | 15+ |
| `providers/` | 图像/运动/安全/状态 Provider 接口 | 4 |
| `skills/` | 技能注册与描述 | 3 |
| `tts/` | 豆包双向流式 TTS + 播放器 | 6 |
| `adapters/` | 适配器基类与注册 | 3 |

### 1.3 评价

- **架构清晰度：★★★★★** — 分层明确，职责边界清晰，contracts 层统一了跨层数据模型
- **模块内聚性：★★★★☆** — 各服务内聚良好，但 `gateways/runtime.py` 存在上帝对象倾向（详见问题清单）
- **可扩展性：★★★★☆** — Provider/Adapter 模式支持多机器人扩展，G1 骨架已存在

## 二、核心功能链路深度检查

### 链路 1：MCP 工具调用链路

**路径：** 外部 Agent → MCP POST `/mcp` → `build_mcp_router` → `GatewayRuntime.invoke_tool` → 具体能力执行 → 返回结果

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 认证鉴权 | ✅ 闭环 | `access_manager.authenticate_http` → `authorize_capability`，支持 Agent/Admin 角色 |
| JSON-RPC 协议 | ✅ 闭环 | initialize / tools/list / tools/call 三个方法完整实现 |
| 错误映射 | ✅ 闭环 | `as_gateway_error` 统一转换异常到 JSON-RPC 错误码 |
| 线程安全 | ✅ 闭环 | `run_in_threadpool` 隔离同步调用 |
| 通知机制 | ⚠️ 缺失 | MCP 协议支持 `notifications/tools/list_changed`，当前 `listChanged: False` 硬编码，无动态通知能力 |
| 批量请求 | ⚠️ 缺失 | JSON-RPC 2.0 规范支持批量请求，当前仅处理单个请求体 |

### 链路 2：导航执行链路

**路径：** `NavigationService` → `Go2GridNavigationPlanner`（A*全局规划）→ `Go2DWALocalPlanner`（DWA局部规划）→ `Go2OfficialBackend`（运动执行）→ Go2 Sport Client

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 全局规划 | ✅ 闭环 | A* 栅格规划器完整实现，含膨胀、最近可通行点搜索 |
| 局部规划 | ✅ 闭环 | DWA 速度采样 + 轨迹评估 + 动态避障 |
| 运动执行 | ✅ 闭环 | 官方后端通过 Sport Client 发送速度命令 |
| 状态机 | ✅ 闭环 | IDLE → PLANNING → FOLLOWING_PATH → FINAL_YAW_ALIGNMENT → ARRIVED，含 DIRECT_FALLBACK 和 RECOVERY |
| 碰撞检测 | ✅ 闭环 | `Go2CollisionDetector` 基于激光雷达，SAFE/WARNING/DANGER 三级 |
| 路径安全评估 | ✅ 闭环 | `evaluate_path_safety` 检查路径与障碍物距离 |
| 路径重采样 | ✅ 闭环 | `resample_waypoints` 统一路径点间距 |
| 导航取消 | ✅ 闭环 | 任务管理器支持取消，安全守卫可强制停止 |
| 代价地图缺失降级 | ⚠️ 部分 | 当代价地图不可用时，DWA 规划器缺少 `cost_map` 参数处理不够健壮，可能退化为纯路径跟踪 |
| 导航超时 | ⚠️ 部分 | 有 `timeout_sec` 参数但缺少导航执行层面的主动超时检测机制，依赖任务管理器层面超时 |

### 链路 3：前沿探索链路

**路径：** `NavigationService` → `Go2FrontierExplorer`（前沿检测）→ `GridNavigationPlanner`（路径规划）→ `Go2OfficialBackend`（运动执行）→ 循环

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 前沿检测 | ✅ 闭环 | 基于占据栅格的 free-unknown 边界检测 |
| 候选排序 | ✅ 闭环 | 综合信息增益、可达距离、代价值评分 |
| 安全检查 | ✅ 闭环 | `check_exploration_safety` 三级安全状态 |
| 探索循环 | ✅ 闭环 | 到达目标 → 重新检测前沿 → 选择下一目标 |
| 已尝试目标记录 | ✅ 闭环 | `attempted_poses` 避免重复探索 |
| 探索终止条件 | ⚠️ 缺失 | 无明确的"探索完成"判定（如无新前沿时自动停止并上报），依赖外部取消 |
| 探索恢复 | ⚠️ 缺失 | 探索过程中如果导航失败，缺少自动恢复策略（如回退、重选目标） |

### 链路 4：建图与地图管理链路

**路径：** `Go2DataPlaneRuntime`（DDS/ROS2 数据源）→ `MappingService` → `MapWorkspaceService`（地图目录管理）→ 持久化存储

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 实时建图 | ✅ 闭环 | DDS 直连 + ROS2 话题双通道，支持全局/局部栅格地图 |
| 地图持久化 | ✅ 闭环 | `save_map_version` / `load_map_version` 版本化管理 |
| 地图激活 | ✅ 闭环 | `activate_map` 切换活跃地图，联动定位和记忆库 |
| 地图删除 | ✅ 闭环 | `delete_map` 含关联记忆库清理选项 |
| 启动恢复 | ✅ 闭环 | `NUWAX_ACTIVE_MAP_RESTORE_ON_START` 自动恢复上次活跃地图 |
| 地图合成 | ✅ 闭环 | 全局稀疏占据 + 局部栅格 + 高程图 + 代价图多源融合 |
| 地图版本回滚 | ✅ 闭环 | 支持按版本 ID 加载历史地图 |
| 地图坐标系一致性 | ⚠️ 风险 | 多源地图融合时，frame_id 一致性依赖 `frame_ids_semantically_equal` 做软匹配，缺少严格坐标变换验证 |

### 链路 5：定位链路

**路径：** `Go2DataPlaneRuntime`（DDS 位姿 / ROS2 里程计）→ `LocalizationService` → 位姿快照

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 实时位姿 | ✅ 闭环 | DDS 直连 `rt/sportmodestate` + ROS2 里程计双通道 |
| 重定位 | ✅ 闭环 | `relocalize_active_map` 支持手动触发 |
| 会话管理 | ✅ 闭环 | `active_session` 跟踪定位状态 |
| 多源里程计 | ✅ 闭环 | `SECONDARY_ODOM_TOPICS` 支持备选里程计源 |
| 定位丢失检测 | ⚠️ 缺失 | 无明确的定位质量评估和丢失检测机制，位姿跳变无告警 |

### 链路 6：感知链路

**路径：** `Go2ProviderBundle`（VideoClient 图像源）→ `PerceptionVideoRuntime`（持续抓帧）→ `PerceptionService`（检测+跟踪+场景摘要）→ 记忆写入

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 本地检测 | ✅ 闭环 | YOLO (Ultralytics) + TensorRT 加速，含 .pt 回退 |
| 云端语义 | ✅ 闭环 | OpenAI 兼容接口，支持视觉场景描述 |
| 元数据检测 | ✅ 闭环 | `MetadataDrivenDetectorBackend` 复用 Go2 自带检测结果 |
| 关键帧策略 | ✅ 闭环 | 基于时间/位移/航向变化的关键帧选择，含抑制逻辑 |
| Burst 复核 | ✅ 闭环 | 到点后短时多帧复核 |
| 目标跟踪 | ✅ 闭环 | `Basic2DTrackerBackend` 简单 2D 跟踪 |
| 场景描述 | ✅ 闭环 | 混合场景描述后端（本地+云端） |
| 失败退避 | ✅ 闭环 | `failure_backoff_sec` 防止连续失败风暴 |
| 感知结果到记忆 | ✅ 闭环 | 关键帧自动写入记忆库 |
| 3D 检测 | ⚠️ 部分 | `detections_3d` 在投影服务中被消费，但 Go2 当前缺少深度相机，3D 检测实际不可用 |
| 感知延迟监控 | ⚠️ 缺失 | 无端到端感知延迟指标采集 |

### 链路 7：记忆链路

**路径：** `PerceptionService` → `VisionToMapProjectionService`（视觉→地图投影）→ `InstanceAssociationService`（实例关联）→ `MemoryRepository` + `SQLiteNamedVectorStore`（持久化）→ 向量检索

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 记忆库生命周期 | ✅ 闭环 | 创建/激活/停用/删除完整 |
| 多模态向量 | ✅ 闭环 | 文本向量（百炼在线/本地 BGE-M3）+ 图像向量（百炼在线/本地 CLIP） |
| 向量回退 | ✅ 闭环 | API Key 缺失时自动回退到 HashingTextEmbedder，图像向量关闭 |
| 语义实例管理 | ✅ 闭环 | 静态/可移动/瞬态三类实例，含生命周期状态机 |
| 空间关联 | ✅ 闭环 | 按距离+属性+时间连续性合并实例 |
| 导航语义上下文 | ✅ 闭环 | `get_navigation_semantic_context` 为导航提供拓扑和锚点信息 |
| 地点标记 | ✅ 闭环 | `tag_location` / `list_tagged_locations` |
| 记忆检索 | ✅ 闭环 | 文本+图像+空间+词法+时间衰减综合评分 |
| 记忆与地图联动 | ✅ 闭环 | 地图激活时自动加载关联记忆库 |
| 向量存储后端 | ✅ 闭环 | SQLite + named vectors，单文件部署 |
| 记忆库隔离 | ✅ 闭环 | 命名记忆库，支持多地图独立记忆 |
| 向量维度一致性 | ⚠️ 风险 | 切换向量模型时，已有向量数据与新模型维度不匹配，缺少自动迁移或告警 |
| 记忆库容量管理 | ⚠️ 缺失 | 无记忆条目上限和淘汰策略，长期运行可能无限增长 |

### 链路 8：音频/TTS 链路

| 中断机制 | ⚠️ 部分 | `interrupt=True` 硬编码，缺少显式的"中断当前播报"API |

### 链路 9：事件流链路

## 三、工程级要素检查

### 3.1 错误处理

| 检查项 | 评级 | 说明 |
|--------|------|------|
| 统一异常体系 | ★★★★★ | `GatewayError` 层次完整，含 HTTP/JSON-RPC 双协议错误码映射 |
| 异常转换 | ★★★★★ | `as_gateway_error` 覆盖 `CapabilityNotRegisteredError`、`ResourceConflictError`、`KeyError` |
| 降级策略 | ★★★★☆ | TTS/感知/向量等关键链路均有降级回退，但部分降级路径缺少日志告警 |
| 异常传播 | ★★★★☆ | 服务层异常大多正确传播，但部分 `except Exception: pass` 吞掉异常（如 `runtime.start` 中的 refresh 调用） |

### 3.2 日志体系

| 检查项 | 评级 | 说明 |
|--------|------|------|
| 结构化日志 | ★★★★★ | `ContextAwareFormatter` 自动追加 extra 字段 |
| 日志限流 | ★★★★★ | `LogRateLimiter` 按时间窗限流重复日志 |
| 运行时事件日志 | ★★★★☆ | 事件同步写入标准日志，但 INFO 级别事件被限流，可能丢失关键信息 |
| 日志轮转 | ★★★★☆ | `NUWAX_LOG_MAX_LINES` + `NUWAX_LOG_TRIM_CHECK_INTERVAL` 支持按行数裁剪 |
| 日志目录 | ★★★☆☆ | 日志目录硬编码为 `/home/unitree/nuwax-robot-bridge`，非 Unitree 环境需手动调整 |

### 3.3 配置管理

| 检查项 | 评级 | 说明 |
|--------|------|------|
| 配置入口 | ★★★★★ | `.env.example` 覆盖全部配置项，含详细中文注释 |
| 环境变量优先级 | ★★★★★ | 环境变量 > `.env` 文件 > 代码默认值 |
| 配置分区 | ★★★★★ | 日志/网关/感知/DDS/TTS/音量/记忆等分区清晰 |
| 兼容性 | ★★★★☆ | 保留旧变量名兼容（如 `GO2_PROXY_LOG_*` → `NUWAX_LOG_*`），但增加了维护负担 |
| 配置校验 | ★★★☆☆ | Pydantic 模型做类型校验，但缺少语义校验（如 IP 地址可达性、模型文件存在性） |

### 3.4 并发与线程安全

| 检查项 | 评级 | 说明 |
|--------|------|------|
| 锁使用 | ★★★★☆ | 关键路径使用 `threading.RLock`，但部分服务（如 `MemoryService`）的复合操作未整体加锁 |
| 线程模型 | ★★★★☆ | 感知运行时/DDS 读取/导航控制各自独立线程，通过 StateStore 解耦 |
| 异步隔离 | ★★★★★ | FastAPI 异步层通过 `run_in_threadpool` 隔离同步调用 |
| 守护线程 | ★★★★☆ | 后台线程均设 `daemon=True`，进程退出时自动终止 |

### 3.6 依赖管理

## 四、问题清单

### 🔴 严重问题（影响功能闭环）

| 编号 | 问题 | 影响范围 | 建议 |
|------|------|----------|------|
| S-01 | **看门狗无定时检查** — `SafetyGuard.check_watchdogs()` 方法存在但无定时调用线程，看门狗机制形同虚设 | 安全 | 在 `GatewayRuntime.start()` 中启动守护线程定期调用 `check_watchdogs()` |
| S-02 | **探索无终止条件** — 前沿探索缺少"无新前沿时自动停止并上报完成"的逻辑，探索只能依赖外部取消 | 探索 | 在探索循环中检测无新前沿候选时，自动设置 `EXPLORATION_COMPLETED` 状态并上报事件 |
