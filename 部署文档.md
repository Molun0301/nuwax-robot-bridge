# 部署文档

本文档用于把 `nuwax_robot_bridge` 从零部署到“实际可用”状态。

按照本文步骤完成后，应达到下面结果：

1. 宿主机上的 `nuwax_robot_bridge` 可以正常启动并开机自启。
2. Docker 容器内的上层客户端可以稳定访问宿主机网关。
3. 容器内的外部 MCP 会自动注入宿主机能力，不依赖手改数据库。
4. 重启宿主机后，宿主机网关、容器和外部 MCP 注入仍然有效。

## 1. 适用部署形态

本文默认场景如下：

1. 机器人为 Go2。
2. `nuwax_robot_bridge` 运行在 Jetson 宿主机。
3. 上层客户端运行在 Docker 容器内。
4. 宿主机与容器通过 `MCP + HTTP + 事件流 + 制品下载` 通信。

部署拓扑如下：

```text
+----------------------------+          +----------------------------------+
| Docker 容器                |          | 宿主机                           |
| 上层客户端                 |          | nuwax_robot_bridge               |
|                            |          |                                  |
| 调用 MCP / HTTP            | -------> | 8766: MCP / HTTP / events        |
| 消费工具结果与制品         | <------- | 连接 Go2 SDK / DDS / 相机 / 音频 |
+----------------------------+          +----------------------------------+
```

职责划分固定如下：

1. 容器负责会话、界面、工具调用和结果消费。
2. 宿主机负责连接机器人并执行真正的具身能力。

## 2. 部署前确认

开始之前请先确认下面几项：

1. 宿主机可以正常连接 Go2。
2. 宿主机当前 Python 解释器可以导入 `unitree_sdk2py`。
3. 宿主机已具备 CycloneDDS 运行时。
4. Docker 已安装并可以正常拉起容器。
5. 你已经拿到上层客户端镜像和启动密钥。

本文假设项目目录为：

```bash
/home/unitree/nuwax-robot-bridge
```

如果你的实际路径不同，请把后续命令中的路径整体替换掉。

## 3. 第一步：准备宿主机代码目录

进入项目目录：

```bash
cd /home/unitree/nuwax-robot-bridge
```

复制配置模板：

```bash
cp .env.example .env
```

## 4. 第二步：安装宿主机依赖

执行：

```bash
cd /home/unitree/nuwax-robot-bridge
python3 -m pip install --user --upgrade "pip>=24.0" setuptools wheel packaging
python3 -m pip install --user --only-binary=:all: "polars==1.8.2"
python3 -m pip install -r requirements.txt
```

如果这里就报错，不要继续后面的步骤，先处理环境问题。最常见的是这三类：

1. `unitree_sdk2py` 当前解释器不可导入。
2. `cyclonedds` 版本与 `unitree_sdk2py` 不匹配。
3. CycloneDDS 原生动态库或依赖库缺失。

如果这里出现 `polars`、`puccinialin` 相关报错，原因通常不是仓库本身，而是宿主机的 `pip` 太旧，没有正确选择 `aarch64` 的预编译 wheel，错误退回了源码构建。上面两条命令就是为了解决这个问题，安装时请始终使用 `python3 -m pip`，不要混用系统里的旧 `pip` 命令。

## 5. 第三步：编辑 `.env`

至少把下面这些配置确认一遍。

### 5.1 Go2 与 DDS

```text
GO2_DDS_IFACE=eth0
GO2_SDK_PATH=/home/unitree/unitree_sdk2_python
CYCLONEDDS_URI=<CycloneDDS><Domain><Compatibility><StandardsConformance>lax</StandardsConformance></Compatibility></Domain></CycloneDDS>
```

说明：

1. `GO2_DDS_IFACE` 必须写实际连接 Go2 的网卡名。
2. `GO2_SDK_PATH` 指向本机可用的 Unitree SDK 路径。
3. 如果你机器上已经通过环境变量设置了别的 `CYCLONEDDS_URI`，以你最终生效的值为准。

### 5.2 网关与鉴权

```text
NUWAX_GATEWAY_HOST=0.0.0.0
NUWAX_GATEWAY_PORT=8766
NUWAX_GATEWAY_PUBLIC_BASE_URL=http://host.docker.internal:8766
NUWAX_GATEWAY_AGENT_TOKENS=agent-robot-bridge-token
NUWAX_GATEWAY_ADMIN_TOKENS=admin-robot-bridge-token
NUWAX_GATEWAY_ALLOWED_SOURCE_CIDRS=127.0.0.1/32,172.16.0.0/12
```

说明：

1. `NUWAX_GATEWAY_PUBLIC_BASE_URL` 直接决定对外生成的制品访问地址。
2. `NUWAX_GATEWAY_AGENT_TOKENS` 要与后面外部 MCP 配置中的 Bearer Token 保持一致。
3. `NUWAX_GATEWAY_ALLOWED_SOURCE_CIDRS` 建议至少覆盖常见 Docker bridge 网段。

### 5.3 日志与运行时数据

```text
NUWAX_LOG_DIR=/home/unitree/nuwax-robot-bridge
NUWAX_LOG_FILE=/home/unitree/nuwax-robot-bridge/nuwax_robot_bridge.log
NUWAX_LOG_LEVEL=INFO
NUWAX_MEMORY_DB_PATH=/home/unitree/nuwax-robot-bridge/runtime_data/memory/vector_memory.db
```

说明：

1. 宿主机网关启动后会同时输出控制台日志和文件日志。
2. 文件日志路径由 `NUWAX_LOG_FILE` 决定。
3. 新部署统一使用 `NUWAX_LOG_*`，旧 `GO2_PROXY_LOG_*` 仅保留兼容回退。

### 5.4 感知配置

```text
NUWAX_PERCEPTION_DEFAULT_DETECTOR_BACKEND=yolo_local
NUWAX_PERCEPTION_YOLO_MODEL=/home/unitree/nuwax-robot-bridge/runtime_data/models/yolo26n.pt
NUWAX_PERCEPTION_YOLO_RUNTIME=tensorrt
NUWAX_PERCEPTION_YOLO_ENGINE_PATH=/home/unitree/nuwax-robot-bridge/runtime_data/models/yolo26n.engine
NUWAX_PERCEPTION_STREAM_ENABLED=true
NUWAX_PERCEPTION_STREAM_AUTO_START=true
NUWAX_PERCEPTION_KEYFRAME_MIN_INTERVAL_SEC=1.0
NUWAX_PERCEPTION_KEYFRAME_MAX_INTERVAL_SEC=5.0
NUWAX_PERCEPTION_BURST_FRAME_COUNT=4
NUWAX_PERCEPTION_BURST_INTERVAL_SEC=0.08
NUWAX_PERCEPTION_OPENAI_ENABLED=false
```

说明：

1. 仓库本身不内置 `yolo26n.pt` 权重文件，你需要在部署阶段把它准备到 `runtime_data/models/`。
2. 当前默认本地视觉链路是 `YOLO26n + TensorRT`，引擎缺失时会自动回退到 `.pt`。
3. 默认持续感知不是“每帧都重检测”，而是“关键帧驱动 + 到点 burst 复核”。
4. 如果你只是先跑通主链路，保持默认即可。
5. 如果云端视觉模型接的是豆包方舟兼容接口，建议把 `NUWAX_PERCEPTION_OPENAI_BASE_URL` 设为 `https://ark.cn-beijing.volces.com/api/v3`，并保持 `NUWAX_PERCEPTION_OPENAI_API_STYLE=auto`；当前版本会自动切到 `responses` 接口。
6. 如果目标环境里的 `openai` SDK 虽然版本不低，但运行时仍然拿不到 `client.responses`，宿主机会自动回退到直接调用方舟 `REST API（表示性状态传输应用程序接口） /responses`，无需手工改代码。
7. 后续再按需启用云端视觉、真实向量化模型和音频链路。

### 5.5 记忆向量模型配置

```text
NUWAX_MEMORY_TEXT_EMBEDDING_MODEL=/home/unitree/nuwax-robot-bridge/runtime_data/models/BAAI__bge-m3
NUWAX_MEMORY_TEXT_EMBEDDING_DIM=1024
NUWAX_MEMORY_IMAGE_EMBEDDING_MODEL=/home/unitree/nuwax-robot-bridge/runtime_data/models/openai__clip-vit-base-patch32
NUWAX_MEMORY_IMAGE_EMBEDDING_DIM=512
NUWAX_MEMORY_VECTOR_STORE_BACKEND=sqlite_named_vectors
```

说明：

1. 新部署默认仍然使用 `BGE-M3` 文本向量和 `CLIP ViT-B/32` 图像向量，但配置值已经改成项目内本地目录。
2. 服务启动阶段不会自动下载模型；如果本地目录不存在或模型不完整，会直接回退到 `hashing-v1` 文本向量并关闭图像向量链路，同时在日志里告警。
3. 只有手工执行下载脚本时才需要访问 Hugging Face（模型托管平台）。
4. 如果你希望默认配置真正生效，不要只改 `.env`，还要安装下面的依赖并先执行模型下载脚本。

Jetson 宿主机推荐先确认 `torch` 可用：

```bash
python3 - <<'PY'
import torch
print(torch.__version__)
print(torch.cuda.is_available())
PY
```

如果输出是 `False`，通常说明你当前装到的是 PyPI 的 CPU 版 `torch`，而不是 Jetson 专用 GPU 版。对于 JetPack 5.1.1 / Python 3.8 这类环境，推荐先替换成 NVIDIA 官方 wheel：

```bash
python3 -m pip uninstall -y torch torchvision torchaudio

python3 -m pip install --user --no-cache-dir \
  "https://developer.download.nvidia.cn/compute/redist/jp/v511/pytorch/torch-2.0.0+nv23.05-cp38-cp38-linux_aarch64.whl"

python3 -m pip install --user --no-cache-dir --no-deps torchvision==0.15.1

python3 - <<'PY'
import torch
print(torch.__version__)
print(torch.version.cuda)
print(torch.cuda.is_available())
print(torch.cuda.device_count())
print(torch.backends.cudnn.is_available())
PY
```

说明：

1. 这组命令已经在 Go2 Jetson / JetPack 5.1.1 / Python 3.8 环境验证通过。
2. 如果你的 JetPack 版本不是 `5.1.1`，不要直接照抄 wheel 地址，应改成对应 JetPack 版本的 NVIDIA 官方 PyTorch wheel。
3. `torchvision==0.15.1` 是与 `torch 2.0.0` 对齐的版本。

确认通过后，再安装语义空间记忆依赖：

```bash
cd /home/unitree/nuwax-robot-bridge
python3 -m pip install -r requirements-memory-models.txt
```

再手工下载默认记忆模型到项目本地目录：

```bash
cd /home/unitree/nuwax-robot-bridge
python3 scripts/prepare_memory_models.py
```

脚本默认会下载：

1. `BAAI/bge-m3` 到 `runtime_data/models/BAAI__bge-m3`
2. `openai/clip-vit-base-patch32` 到 `runtime_data/models/openai__clip-vit-base-patch32`

如果你想使用其他输出目录，可以显式指定：

```bash
cd /home/unitree/nuwax-robot-bridge
python3 scripts/prepare_memory_models.py \
  --text-output-dir /data/models/bge-m3 \
  --image-output-dir /data/models/clip-vit-base-patch32
```

然后把对应路径写回 `.env`：

```text
NUWAX_MEMORY_TEXT_EMBEDDING_MODEL=/data/models/bge-m3
NUWAX_MEMORY_IMAGE_EMBEDDING_MODEL=/data/models/clip-vit-base-patch32
```

说明：

1. 你也可以继续把环境变量配置成 Hugging Face 仓库名，例如 `sentence-transformers:BAAI/bge-m3`；但启动阶段仍然不会自动下载，只有本地缓存已经准备好时才会直接加载。
2. 如果部署环境无法访问 Hugging Face，请在可联网机器上先执行脚本下载，再把整个模型目录拷贝到目标宿主机。
3. 推荐优先使用项目内本地目录，这样部署行为更可控，排障时也更直观。

### 5.6 Go2 官方定位、地图与探索

这部分配置不再放进平台根级设置逻辑里，而是由 Go2 端侧专属模块负责解析。

配置示例文件在：

```text
drivers/robots/go2/data_plane.env.example
```

推荐实际部署时复制一份到：

```text
drivers/robots/go2/data_plane.env
```

环境变量优先级如下：

1. shell / `systemd` 导出的环境变量
2. `drivers/robots/go2/data_plane.env`
3. 项目根目录 `.env`

常用项如下：

```text
GO2_DATA_PLANE_ENABLED=true
GO2_DATA_PLANE_SETUP_SCRIPT=/home/unitree/nuwax-robot-bridge/drivers/robots/go2/ros2_ws/install/setup.bash
GO2_DATA_PLANE_DDS_IFACE=eth0
GO2_DATA_PLANE_ODOM_TOPIC=/utlidar/robot_odom
GO2_DATA_PLANE_SECONDARY_ODOM_TOPICS=/uslam/localization/odom,/lio_sam_ros2/mapping/odometry,/odom_lio
GO2_DATA_PLANE_IMU_TOPIC=/utlidar/imu
GO2_DATA_PLANE_POINT_CLOUD_TOPIC=/utlidar/cloud
GO2_DATA_PLANE_TF_TOPIC=/tf
GO2_DATA_PLANE_TF_STATIC_TOPIC=/tf_static
GO2_DATA_PLANE_OCCUPANCY_TOPIC=/map
GO2_DATA_PLANE_GRID_MAP_TOPIC=/elevation_mapping_cupy/elevation_map_raw
GO2_DATA_PLANE_COST_MAP_TOPIC=/global_costmap/costmap
GO2_DATA_PLANE_GLOBAL_MAP_RESOLUTION_M=0.05
GO2_DATA_PLANE_GLOBAL_MAP_INFLATION_RADIUS_M=0.05
GO2_DATA_PLANE_LOCAL_MAP_RESOLUTION_M=0.05
GO2_DATA_PLANE_LOCAL_MAP_INFLATION_RADIUS_M=0.05
GO2_NAVIGATION_PLANNER_INFLATION_RADIUS_M=0.05
GO2_COST_MAPPER_RESOLUTION_M=0.05
GO2_COST_MAPPER_OBSTACLE_INFLATION_RADIUS_M=0.05
GO2_VOXEL_MAPPER_VOXEL_SIZE=0.05

GO2_OFFICIAL_BACKEND_ENABLED=true
GO2_OFFICIAL_SERVICES_AUTO_START=false
GO2_OFFICIAL_SERVICE_NAMES=unitree_lidar,unitree_lidar_slam,voxel_height_mapping,obstacles_avoid
GO2_OFFICIAL_CLIENT_TIMEOUT_SEC=3.0
GO2_OFFICIAL_OBSTACLE_AVOIDANCE_DEFAULT_ENABLED=true

GO2_EXPLORATION_ENABLED=true
```

说明：

1. `GO2_DATA_PLANE_ENABLED=true` 后，Go2 装配入口会把官方定位、地图、探索能力接入统一 provider。
2. Go2 现在不再走 `/dev/ttyUSB0` 串口雷达链，也不再依赖本地 Nav2。
3. 默认定位主话题是 `/utlidar/robot_odom`，其余里程计话题仅作为回退订阅源。
4. `/map`、`grid_map`、`cost_map` 只要官方服务启动后出现，平台就会直接消费；不会再尝试本地拉起 Point-LIO 或 Nav2。
5. `GO2_OFFICIAL_SERVICES_AUTO_START=false` 时，网关启动只准备入口，不主动拉起 Go2 官方建图服务；在显式探索或启用记忆库时才按需启用 `unitree_lidar`、`unitree_lidar_slam`、`voxel_height_mapping`。
6. `GO2_OFFICIAL_OBSTACLE_AVOIDANCE_DEFAULT_ENABLED=true` 表示一旦 `obstacles_avoid` 服务被显式启用，默认避障开关保持开启；它不再要求网关启动时立即拉起该服务。
7. 探索后端复用官方运动控制和官方里程计，不再建立本地 `cmd_vel + Nav2` 链。

## 6. 第四步：构建 Go2 端侧 ROS2 工作空间

先准备 Go2 端侧专属配置：

```bash
cd /home/unitree/nuwax-robot-bridge
cp drivers/robots/go2/data_plane.env.example drivers/robots/go2/data_plane.env
```

后续 `drivers/robots/go2/launch_ros2_stack.sh` 会优先读取这份文件，不会直接 `source` 项目根目录 `.env`。
当前网关只会读取这份文件中的官方链路配置，不会再尝试旧的串口雷达和本地 Nav2 启动项。

然后构建仓库内置的 Go2 ROS2 工作空间：

```bash
cd /home/unitree/nuwax-robot-bridge
bash drivers/robots/go2/build_ros2_workspace.sh
```

这个工作空间现在只保留平台需要的最小 ROS2 环境，不再承载串口雷达、Point-LIO 和本地 Nav2。

如果这里构建失败，优先检查：

1. ROS2 本体是否已安装
2. Go2 宿主机上的基础 ROS2 Python 依赖是否齐全

## 7. 第五步：准备 YOLO26n 权重并为 Jetson 导出 TensorRT 引擎

如果你想直接整段复制执行，可以使用下面这组命令：

```bash
cd /home/unitree/nuwax-robot-bridge
mkdir -p runtime_data/models

python3 scripts/prepare_yolo_weights.py \
  --model yolo26n.pt \
  --output /home/unitree/nuwax-robot-bridge/runtime_data/models/yolo26n.pt

python3 scripts/export_yolo_tensorrt_engine.py \
  --weights /home/unitree/nuwax-robot-bridge/runtime_data/models/yolo26n.pt \
  --output /home/unitree/nuwax-robot-bridge/runtime_data/models/yolo26n.engine \
  --imgsz 640 \
  --device 0 \
  --half
```

先创建模型目录：

```bash
cd /home/unitree/nuwax-robot-bridge
mkdir -p runtime_data/models
```

然后准备 `yolo26n.pt`。推荐直接使用仓库内脚本，通过 Ultralytics 官方下载逻辑把权重固化到本地：

```bash
cd /home/unitree/nuwax-robot-bridge
python3 scripts/prepare_yolo_weights.py \
  --model yolo26n.pt \
  --output /home/unitree/nuwax-robot-bridge/runtime_data/models/yolo26n.pt
```

说明：

1. 这一步需要宿主机能联网访问 Ultralytics 官方模型资源。
2. 如果你的部署环境不能联网，就手工把 `yolo26n.pt` 放到 `/home/unitree/nuwax-robot-bridge/runtime_data/models/yolo26n.pt`。
3. 后续平台默认读取的就是这个本地路径，不依赖仓库自带权重。

如果你希望真正走推荐生产链路，而不是让平台回退到 `.pt`，再继续导出 `yolo26n.engine`。

执行：

```bash
cd /home/unitree/nuwax-robot-bridge
python3 scripts/export_yolo_tensorrt_engine.py \
  --weights /home/unitree/nuwax-robot-bridge/runtime_data/models/yolo26n.pt \
  --output /home/unitree/nuwax-robot-bridge/runtime_data/models/yolo26n.engine \
  --imgsz 640 \
  --device 0 \
  --half
```

说明：

1. 该脚本调用 Ultralytics 官方导出能力，底层格式为 `engine`。
2. Jetson 侧 TensorRT 依赖由 JetPack 提供，不通过 `pip` 安装。
3. 如果你的设备不支持当前导出参数，可以保留 `.pt` 回退路径先启动平台。
4. 如果没有导出 `engine`，平台仍可使用上一步准备好的 `.pt` 启动本地检测，但性能会低于 TensorRT 路径。

## 9. 第七步：再手工启动宿主机网关

先不要急着做开机自启，也不要先起容器。

确认官方里程计和官方地图话题状态正常后，再启动宿主机网关：

```bash
cd /home/unitree/nuwax-robot-bridge
source /home/unitree/nuwax-robot-bridge/drivers/robots/go2/source_runtime_env.sh
python3 -m uvicorn gateways.app:create_gateway_app_from_env --factory --host 0.0.0.0 --port 8766
```

说明：

1. 如果启用了 Go2 ROS2 数据面，建议手工启动和 `systemd` 启动都先加载 `source_runtime_env.sh`。
2. 这个脚本会继续加载 ROS2 工作空间的 `setup.bash`，并把兼容 `unitree_sdk2py` 的 CycloneDDS 动态库目录放到 `LD_LIBRARY_PATH` 前面。
3. 这样可以同时规避 `librclpy_common.so` 一类动态库缺失，以及 `ddsi_sertype_v0` 这类 CycloneDDS 版本不匹配问题。

再开一个终端做健康检查：

```bash
curl http://127.0.0.1:8766/api/health \
  -H 'Authorization: Bearer agent-robot-bridge-token'
```

这一步必须先通过，后面容器联通和 MCP 注入才有意义。

如果启动失败，优先检查：

1. `GO2_DDS_IFACE` 是否正确。
2. `unitree_sdk2py` 是否可导入。
3. `CYCLONEDDS_URI` 是否被别的环境变量覆盖。
4. `.env` 中的 token 和网段配置是否正确。

启动后可直接查看平台文件日志：

```bash
tail -f /home/unitree/nuwax-robot-bridge/nuwax_robot_bridge.log
```

## 10. 第八步：配置宿主机开机自启

当第四步已经通过，再配置 `systemd`。

先准备日志和制品目录：

```bash
sudo mkdir -p /var/log/nuwax_robot_bridge
sudo mkdir -p /var/lib/nuwax_robot_bridge/artifacts
sudo chown -R unitree:unitree /var/log/nuwax_robot_bridge
sudo chown -R unitree:unitree /var/lib/nuwax_robot_bridge
```

安装服务文件：

```bash
sudo cp /home/unitree/nuwax-robot-bridge/deploy/systemd/nuwax-host-gateway.service.example \
  /etc/systemd/system/nuwax-host-gateway.service
```

如果你的宿主机用户不是 `unitree`，先编辑：

```bash
sudo nano /etc/systemd/system/nuwax-host-gateway.service
```

至少确认下面几项：

1. `User`
2. `Group`
3. `WorkingDirectory`
4. `ExecStart`

当前仓库内的 `deploy/systemd/nuwax-host-gateway.service.example` 已经默认在 `ExecStart` 中先尝试加载：

```text
/home/unitree/nuwax-robot-bridge/drivers/robots/go2/source_runtime_env.sh
```

如果你的实际部署路径不同，记得同步修改这一段。

然后加载并启动：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now nuwax-host-gateway
sudo systemctl status nuwax-host-gateway
```

实时看日志：

```bash
journalctl -u nuwax-host-gateway -f
```

如果你希望看平台自己的业务日志，而不是只看 `systemd` 输出，则查看：

```bash
tail -f /home/unitree/nuwax-robot-bridge/nuwax_robot_bridge.log
```

## 11. 第九步：启动上层客户端容器

如果你使用的是 Docker 直接运行方式，推荐命令如下：

```bash
docker run -d \
  --name nuwax-client \
  --restart always \
  --add-host=host.docker.internal:host-gateway \
  -p 6080:6080 \
  -v nuwax_home:/home/user \
  -e TZ=Asia/Shanghai \
  -e NUWAX_SAVED_KEY=替换成你的个人密钥 \
  nuwax-docker-images-registry.cn-hangzhou.cr.aliyuncs.com/nuwax/nuwax-agent-client:latest
```

这里最关键的是：

```text
--add-host=host.docker.internal:host-gateway
```

没有这一项，容器通常无法通过 `host.docker.internal` 访问宿主机。

如果你之前已经起过同名容器，但没有带这项参数，建议先删掉重建：

```bash
docker rm -f nuwax-client
```

## 10. 第八步：先验证容器到宿主机网络

不要先猜外部 MCP 是否生效，先验证容器能否访问宿主机健康接口。

在宿主机执行：

```bash
curl http://127.0.0.1:8766/api/health \
  -H 'Authorization: Bearer agent-robot-bridge-token'
```

在容器执行：

```bash
docker exec -it nuwax-client sh -lc \
  "curl http://host.docker.internal:8766/api/health -H 'Authorization: Bearer agent-robot-bridge-token'"
```

如果宿主机返回正常而容器不正常，优先检查：

1. 容器是否真的带了 `--add-host=host.docker.internal:host-gateway`。
2. `.env` 中的 `NUWAX_GATEWAY_ALLOWED_SOURCE_CIDRS` 是否覆盖当前 Docker 网段。
3. Bearer Token 是否与 `.env` 中一致。

## 11. 第九步：安装外部 MCP 注入

当前正式做法不是手改数据库，而是使用仓库自带的外部 MCP 注入脚本。

在宿主机执行：

```bash
cd /home/unitree/nuwax-robot-bridge
sudo python3 scripts/nuwaclaw_external_mcp_hook.py install --init-config
```

这一步会完成两件事：

1. 给容器内的 `nuwax-mcp-stdio-proxy` 安装一层包装。
2. 在宿主机持久化目录生成外部 MCP 配置文件。

默认生成的配置文件路径是：

```text
/var/lib/docker/volumes/nuwax_home/_data/.nuwaxbot/mcp/external_servers.json
```

默认内容应为：

```json
{
  "mcpServers": {
    "nuwax_robot_bridge": {
      "url": "http://host.docker.internal:8766/mcp",
      "headers": {
        "Authorization": "Bearer agent-robot-bridge-token"
      }
    }
  }
}
```

如果你的宿主机端口、token 或容器卷名不同，请在这里改成真实值。

安装后查看状态：

```bash
cd /home/unitree/nuwax-robot-bridge
sudo python3 scripts/nuwaclaw_external_mcp_hook.py status
```

然后重启容器，使注入逻辑重新生效：

```bash
sudo docker restart nuwax-client
```

## 12. 第十步：验证外部 MCP 已可用

这一步的目标不是只看网关健康，而是确认容器里的外部 MCP 已经能把请求转发到宿主机。

建议同时做下面三项：

### 11.1 看宿主机网关日志

```bash
journalctl -u nuwax-host-gateway -f
```

### 11.2 再次确认注入状态

```bash
cd /home/unitree/nuwax-robot-bridge
sudo python3 scripts/nuwaclaw_external_mcp_hook.py status
```

### 11.3 在上层客户端里触发一次机器人相关调用

推荐从低风险、只读能力开始，例如：

1. 查询机器人状态。
2. 查询能力列表。
3. 抓取当前图像。

只要宿主机网关日志里出现对应请求，并且上层客户端拿到了返回结果，就说明部署已经进入“实际可用”状态。

## 13. 第十一步：验证开机自启链路

部署完成后，不要只满足于“当前能跑”，还要确认“重启后还能跑”。

重启宿主机后，执行下面检查：

```bash
systemctl status nuwax-host-gateway
docker ps | grep nuwax-client
docker exec -it nuwax-client sh -lc \
  "curl http://host.docker.internal:8766/api/health -H 'Authorization: Bearer agent-robot-bridge-token'"
cd /home/unitree/nuwax-robot-bridge && \
sudo python3 scripts/nuwaclaw_external_mcp_hook.py status
```

如果这几项都正常，说明整个开机自启链路已经成立。

## 14. 常见问题

### 13.1 宿主机健康检查返回 `401`

通常是 token 不一致。

检查：

1. `.env` 中的 `NUWAX_GATEWAY_AGENT_TOKENS`
2. `external_servers.json` 中的 `Authorization`

### 13.2 容器访问宿主机返回 `403`

通常是来源网段不在白名单里。

检查：

```text
NUWAX_GATEWAY_ALLOWED_SOURCE_CIDRS
```

调试阶段建议至少覆盖：

```text
127.0.0.1/32,172.16.0.0/12
```

### 13.3 容器里访问不到 `host.docker.internal`

通常是容器没有带：

```text
--add-host=host.docker.internal:host-gateway
```

### 13.4 Go2 启动时报 DDS 初始化失败

优先检查：

1. `GO2_DDS_IFACE` 是否写成了真实网卡。
2. 宿主机当前解释器是否真能导入 `unitree_sdk2py`。
3. `CYCLONEDDS_URI` 是否被宿主机其他环境变量覆盖。
4. `cyclonedds` 与 `unitree_sdk2py` 的版本是否匹配。
5. 当前启动命令是否先 `source drivers/robots/go2/source_runtime_env.sh`，并让兼容版本的 `libddsc.so` 排在 ROS Foxy 自带库前面。

### 13.5 外部 MCP 注入装上了，但容器重启后看起来没效果

优先检查：

1. `sudo python3 scripts/nuwaclaw_external_mcp_hook.py status`
2. `/var/lib/docker/volumes/nuwax_home/_data/.nuwaxbot/mcp/external_servers.json`
3. 容器是否已经执行过 `docker restart nuwax-client`
4. 外部 MCP JSON 中的 URL 和 token 是否仍然正确

第一次部署成功后，建议立刻做三件事：

1. 备份 `.env`
2. 备份 `external_servers.json`
3. 记录当前网关端口、token、容器名和 `systemd` 服务名
